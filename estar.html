C&DH

High level, low level efforts
On this subsystem I worked on the software side:
  • Operative modes definition and implementation, including scheduling, mode switching from telecommands, management of the activated/deactivated equipments in each mode
  • Telecommands implementation, both simple C&DH ones such as telemetry management, antenna deployment, mission time update, and so on, and cross-subsystem ones, such as telecommands for the ADCS, such as updating UTC time (used for the propagator), changing the torquers' polarity, and more.
  • Implementation of functions to help during the ground tests, such has interfacing the C&DH with the USB port.

Things learned:
MSP430 architecture, SD, serial and USB interfaces, data driven telecommands and telemetry implementation, automated testing and testing procedures, low level programming such as pin commanding and reading, telemetry packet building, scheduling and real time operations (and threads, semaphores, custom libraries from the vendor).

ADCS

From component tests to algorithm implementation

Regarding the ADCS of the two CubeSats, I participated in the design and development during the second CubeSat.
I was participating in several activities:

  • IMU testing
  • Magnetic torquers characterization and functional testing
  • Development of complete flight code in C. Portions of it were realized with the participation of my colleague Raffaele.
  • Definition of the high level architecture, and the interfaces between ADCS and C&DH systems.
  • Algorithms development: B-dot control, Y-Thomson control
  • Testing and verification of the implemented algorithm with comparisons with Matlab/Simulink code and STK simulations.
Things learned:
ARM v9 architecture, RT Linux, sensor characterization and functional testing (IMU), actuator functional testing (Magnetic torquers), code verification with STK (SGP4 propagator), magnetic field model implementation and verification via Matlab/Simulink, basics of Kalman Filtering, controls (B-Dot, Y-Thomson for detumbling), PWM control, IMU sampling (gyros, magnetometers) and more.

System Engineering

Grasping the full picture!

For e-st@r-II, it was adopted a mixture of document-based and SysML-based approaches. Here is a list of all the activities and documents I have dealt with at a system level:

  • CubeSat Design Description, it describes into the details the e-st@r-II CUbeSat, from the system point of view
  • Environmental Test Plan, it aims at providing information about the planning of the environmental tests to be performed on e-st@r-II CubeSat. The document guides the developer through the definition of the vibration and thermal-vacuum tests, giving information about the GSE and test facilities to be used, the personnel, organisation and management of the environmental test campaign and its schedule
  • Environmental Test Procedure, it aims at providing all the information required to fully define each test activity foreseen in Environmental Test Plan for the environmental test campaign. In particular, the requirements to be verified, the test approach and requirements for each related acitivity, the precise set-up, the GSE and pass/fail criteria for each test are described.
  • FDS-VCD, it is the Verification Control Document for Fly Your Satellite! Design Specifications. It defines, in matrix form, for each requirement to be verified, the combination of the selected verification methods for the different verification levels at the applicable verification stages.
  • Functional Test Reports. One for each basic and full functional test performed during Fly Your Satellite! programme. It derives the information contained in the Test Plan, Specification and Procedure document, and includes additional elements such as the completed as-run procedure.
  • Memo, several memorandum were produced to provide to ESA specific information, ranging from info at component level up to system level, in a more concise way.
  • Mission Description Document, it is aimed to describe the e-st@r-II mission. The purpose is to give a description of the mission, to provide input for the selection of the best architecture meeting the mission statement in iteration with the definition of the system specification. It presents how the objectives, operation profile, major system events and capabilities, contingencies and performance are expected to be achieved.The document structure goes through the definition of the mission requirements and mission architecture. Then a description of the mission scenario is provided, in terms of main mission phases, satellite operational modes, nominal orbit activities and contingencies, from the perspective of space and ground segments
  • Mission Test Report, it is structured in a similar way to functional test reports, but it provides information about the mission test
  • NCR, during the verification campaigns, several non-conformance occurred. A non-conformance report documents the details of a non-conformance identified in a quality audit or other process review. The objective of the report is to make an unambiguous, defensible, clear and concise definition of the problem so that corrective action can and will be initiated by management
  • Request For Waiver, the objective of the RFW is to be the vehicle for requiring and agreeing to the use or the delivery of a product that does not conform to its approved product configuration baseline
  • Review Item Discrepancy, more than 100 RIDs were created in FYS programme. RIDs are prepared to record identified problems and questions arising from the examination of the review documentation
  • Space Debris Mitigation Report, it reports the assessments, justifications and verification of compliance with ESA requirements about space debris regulations
  • SSD-VCD, it is the VCD for system specifications. It defines, in matrix form, for each requirement to be verified, the combination of the selected verification methods for the different verification levels at the applicable verification stages
  • Test Report, several test reports were produced. They range from tests at component level up to system level. They include about the test set-up, GSE used, as-run procedures and result analysis

AIV

In the loop simulations

I've worked to develop a Hardware-In-The-Loop simulator as my MSc thesis. The simulator was developed in Python (even though the flight code was in C) and included features as automatic code generation, simulation architecture selection (Algorithm-, Software-, Controller- and Hardware-In-The-Loop). The simulator was presented several congresses; download the most recent presentation here.

Additional activities:

  • Assembly and verification of e-st@r-II CubeSat
  • Functional test campaign at Polito
  • Integration campaign at CSG, Kourou, French Guiana
