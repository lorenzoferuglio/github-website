C&DH

High level, low level efforts

My contribution on e-st@r I C&DH system has been limited: the development was pretty much completed already when I joined the team, so all I did was debug the code and test it.
e-st@r II, on the other side, saw me deeply involved in the development: taking as a base the heritage of e-st@r I, I worked to completely change the behavior of three operative modes, and several telecommands were improved or rewritten from scratch. Furthermore, I introduced new features that allowed easier execution of test procedures during functional testing, for example the possibility of controlling the satellite via USB.

ADCS

From component tests to algorithm implementation

Regarding the ADCS of the two CubeSats, I participated in the design and development during the second CubeSat.
I was participating in several activities:

  • IMU testing
  • Responsible for the implementation of the flight code in C. Portions of it were realized with the participation of a colleague.
  • Definition of the high level architecture, and the interfaces between ADCS and C&DH systems.
  • Testing and verification of the implemented algorithm with comparisons with Matlab/Simulink code and STK simulations.

AIV

In the loop simulations

I've worked to develop a Hardware-In-The-Loop simulator as my MSc thesis. The simulator was developed in Python (even though the flight code was in C) and included features as automatic code generation, simulation architecture selection (Algorithm-, Software-, Controller- and Hardware-In-The-Loop). The simulator was presented several congresses; download the most recent presentation here.
